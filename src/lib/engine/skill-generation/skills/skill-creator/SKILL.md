# 技能创建器 (Skill Creator)

用于创建新技能并对其进行迭代改进的技能。

从高层次来看，创建技能的过程如下：

- 确定你希望技能做什么以及大致如何实现。
- 编写技能草案。
- 创建一些测试提示词，并在其上运行具备该技能访问权限的 Claude。
- 协助用户进行定性和定量的结果评估。
  - 在后台运行测试时，如果没有定量评测（evals），请草拟一些（如果已有，可以直接使用或根据需要进行修改）。然后向用户解释这些评测（或者解释现有的评测）。
  - 使用 `eval-viewer/generate_review.py` 脚本向用户展示结果供其查看，并让他们查看定量指标。
- 根据用户评估结果的反馈（以及定量基准测试中发现的明显缺陷）重写技能。
- 重复此过程直到满意。
- 扩大测试集并尝试更大规模的测试。

在使用此技能时，你的任务是弄清楚用户处于该过程的哪个阶段，然后介入并帮助他们推进。例如，如果用户说“我想做一个关于 X 的技能”，你可以帮助缩小其含义、编写草案、编写测试用例、确定评估方式、运行所有提示词并重复。

另一方面，如果他们已经有了技能草案，你可以直接进入循环中的评测/迭代部分。

当然，你应该始终保持灵活。如果用户说“我不需要运行一堆评估，直接凭感觉来”，你也可以照做。

在技能完成后（顺序是灵活的），你还可以运行技能描述优化器（我们有一个专门的独立脚本），以优化技能的触发。

明白了吗？好的。

## 与用户沟通

技能创建器可能会被各种编码背景的人使用。如果你还没听说（其实也是最近才开始的趋势），Claude 的强大正在激励水管工打开终端，激励家长和祖父母去谷歌搜索“如何安装 npm”。另一方面，大部分用户可能具备相当程度的计算机素养。

因此，请注意上下文线索以理解如何组织你的语言！在默认情况下，给你一些参考：

- “评估（evaluation）”和“基准测试（benchmark）”处于边缘，但可以使用。
- 对于 “JSON” 和 “断言（assertion）”，在不加解释地使用它们之前，你需要从用户那里看到明显的信号，证明他们知道这些是什么。

如果不确定，简要解释术语是可以的；如果你不确定用户是否理解，可以随时提供简短的定义来澄清术语。

**输出格式：**
- 如果你的输出包含代码或 JSON 数据，必须使用 Markdown 代码块（例如 ```json\n...\n``` 或 ```python\n...\n```）进行格式化，以便前端显示。

**语言要求：**
- 除非用户明确要求使用其他语言，否则对于所有面向用户的自然语言文本，请使用中文回复。

## 创建技能

### 捕获意图

首先理解用户的意图。当前的对话可能已经包含用户想要捕获的工作流（例如，他们说“把这个变成技能”）。如果是这样，先从对话历史中提取答案 —— 使用的工具、步骤序列、用户所做的更正、观察到的输入/输出格式。用户可能需要填补空白，并在继续下一步之前进行确认。

1. 这个技能应该让 Claude 能够做什么？
2. 这个技能应该在什么时候触发？（哪些用户短语/上下文）
3. 预期的输出格式是什么？
4. 我们是否应该设置测试用例来验证技能是否工作？具有客观可验证输出（文件转换、数据提取、代码生成、固定工作流步骤）的技能会从测试用例中受益。具有主观输出（写作风格、艺术）的技能通常不需要。根据技能类型建议合适的默认设置，但让用户决定。

### 访谈与研究

主动询问关于边界情况、输入/输出格式、示例文件、成功标准和依赖项的问题。在处理好这部分之前，不要急于编写测试提示词。

检查可用的 MCP —— 如果对研究有用（搜索文档、寻找类似技能、查找最佳实践），如果可用，通过子 Agent 并行研究，否则内联研究。带着上下文来，以减轻用户的负担。

### 编写 SKILL.md

基于用户访谈，填充这些组件：

- **name**: 技能标识符
- **description**: 何时触发，它做什么。这是主要的触发机制 —— 包含技能的功能以及使用的具体上下文。所有的“何时使用”信息都放在这里，而不是正文中。注意：目前 Claude 倾向于“触发不足” —— 即在技能有用时不去使用。为了应对这一点，请让技能描述稍微带一点“强制性”。例如，不要写“如何构建一个简单的快速仪表盘来显示内部 Anthropic 数据”，而可以写成“如何构建一个简单的快速仪表盘来显示内部 Anthropic 数据。每当用户提到仪表盘、数据可视化、内部指标或想要显示任何类型的公司数据时，即使他们没有明确要求‘仪表盘’，也请务必使用此技能。”
- **compatibility**: 所需工具、依赖项（可选，很少需要）
- **技能的其余部分 :)**

### 技能编写指南

#### 技能结构 (Anatomy of a Skill)

```
skill-name/
├── SKILL.md (必需)
│   ├── YAML 前置元数据 (name, description 必需)
│   └── Markdown 指令
└── 捆绑资源 (可选)
    ├── scripts/    - 用于确定性/重复性任务的可执行代码
    ├── references/ - 根据需要加载到上下文中的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

#### 渐进式加载 (Progressive Disclosure)

技能使用三层加载系统：
1. **元数据** (name + description) - 始终在上下文中（~100 字）
2. **SKILL.md 正文** - 技能触发时进入上下文（理想情况下 <500 行）
3. **捆绑资源** - 按需加载（无限制，脚本可以执行而无需加载）

这些字数是近似值，如果需要，你可以写得更长。

**关键模式：**
- 保持 SKILL.md 在 500 行以内；如果你快要达到这个限制，请增加额外的层级，并提供清晰的指针，说明模型下一步应该去哪里。
- 在 SKILL.md 中清晰地引用文件，并指导何时阅读它们。
- 对于大型参考文件（>300 行），包含一个目录。

**领域组织**：当技能支持多个领域/框架时，按变体组织：
```
cloud-deploy/
├── SKILL.md (工作流 + 选择)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claude 仅读取相关的参考文件。

#### 无惊喜原则 (Principle of Lack of Surprise)

不用说，技能不得包含恶意软件、漏洞利用代码或任何可能损害系统安全的内容。技能的内容在描述意图时不应让用户感到意外。不要配合创建误导性技能或旨在促进未经授权的访问、数据外泄或其他恶意活动的技能。不过，像“扮演 XYZ”之类的角色扮演是可以的。

#### 编写模式

在指令中优先使用祈使句。

**定义输出格式** - 你可以这样做：
```markdown
## 报告结构
始终使用此确切模板：
# [标题]
## 执行摘要
## 关键发现
## 建议
```

**示例模式** - 包含示例非常有用。你可以这样格式化（但如果示例中包含“输入”和“输出”，你可能需要稍微偏离一下）：
```markdown
## 提交消息格式
**示例 1：**
输入：Added user authentication with JWT tokens
输出：feat(auth): implement JWT-based authentication
```

### 编写风格

尝试向模型解释为什么某些事情很重要，而不是使用生硬的“必须（MUST）”。使用心理理论（theory of mind），并尝试使技能具有通用性，而不是局限于特定示例。先写一个草案，然后以全新的视角审视并改进它。

### 测试用例

编写完技能草案后，构思 2-3 个真实的测试提示词 —— 即真实用户实际会说的话。与用户分享：[你不必使用完全相同的语言] “这里有几个我想尝试的测试用例。这些看起来正确吗，或者你想添加更多？” 然后运行它们。

将测试用例保存到 `evals/evals.json`。先不要写断言 —— 只写提示词。你将在下一步中在运行进行期间起草断言。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示词",
      "expected_output": "预期结果的描述",
      "files": []
    }
  ]
}
```

有关完整架构（包括你稍后将添加的 `assertions` 字段），请参见 `references/schemas.md`。

## 运行并评估测试用例

本节是一个连续的序列 —— 不要中途停止。不要使用 `/skill-test` 或任何其他测试技能。

将结果放在 `<skill-name>-workspace/` 中，作为技能目录的同级目录。在工作区内，按迭代组织结果（`iteration-1/`, `iteration-2/` 等），在其中，每个测试用例都有一个目录（`eval-0/`, `eval-1/` 等）。不要预先创建所有这些目录 —— 边进行边创建。

### 第 1 步：在同一轮中启动所有运行（有技能运行和基准运行）

对于每个测试用例，在同一轮中启动两个子 Agent —— 一个带有技能，一个不带。这很重要：不要先启动有技能的运行，然后再回来运行基准测试。一次性启动所有任务，以便它们大约在同一时间完成。

**有技能运行：**

```
执行此任务：
- 技能路径: <技能路径>
- 任务: <评测提示词>
- 输入文件: <评测文件（如果有），否则为 "无">
- 保存输出至: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 要保存的输出: <用户关心的内容 —— 例如，“.docx 文件”，“最终的 CSV”>
```

**基准运行**（相同的提示词，但基准取决于上下文）：
- **创建新技能**：完全不带技能。相同的提示词，无技能路径，保存到 `without_skill/outputs/`。
- **改进现有技能**：旧版本。在编辑之前，对技能进行快照（`cp -r <skill-path> <workspace>/skill-snapshot/`），然后让基准子 Agent 指向该快照。保存到 `old_skill/outputs/`。

为每个测试用例编写一个 `eval_metadata.json`（断言暂时可以为空）。根据测试内容给每个评测起一个描述性名称 —— 不仅仅是 “eval-0”。该名称也用于目录。如果本次迭代使用了新的或修改过的评测提示词，请为每个新的评测目录创建这些文件 —— 不要假设它们会从之前的迭代中继承。

```json
{
  "eval_id": 0,
  "eval_name": "此处填描述性名称",
  "prompt": "用户的任务提示词",
  "assertions": []
}
```

### 第 2 步：在运行进行期间，起草断言

不要只是等待运行结束 —— 你可以高效地利用这段时间。为每个测试用例起草定量断言并向用户解释。如果 `evals/evals.json` 中已经存在断言，请审查并解释它们检查的内容。

好的断言是客观可验证的，并且具有描述性名称 —— 它们在基准测试查看器中应该易于阅读，以便有人扫一眼结果就能立即理解每个断言检查的内容。主观技能（写作风格、设计质量）最好进行定性评估 —— 不要强行在需要人类判断的事情上使用断言。

起草完断言后，更新 `eval_metadata.json` 文件和 `evals/evals.json`。还要向用户解释他们将在查看器中看到什么 —— 包括定性输出和定量基准。

### 第 3 步：随着运行完成，捕获计时数据

当每个子 Agent 任务完成时，你会收到包含 `total_tokens` 和 `duration_ms` 的通知。立即将这些数据保存到运行目录中的 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是捕获此数据的唯一机会 —— 它通过任务通知传递，不会持久保存在其他地方。在每个通知到达时进行处理，而不是尝试批量处理。

### 第 4 步：评分、聚合并启动查看器

所有运行完成后：

1. **为每次运行评分** —— 启动一个评分员子 Agent（或内联评分），读取 `agents/grader.md` 并根据输出评估每个断言。将结果保存到每个运行目录中的 `grading.json`。`grading.json` 的 `expectations` 数组必须使用 `text`, `passed`, 和 `evidence` 字段（而不是 `name`/`met`/`details` 或其他变体） —— 查看器依赖于这些确切的字段名称。对于可以通过编程检查的断言，编写并运行脚本而不是目测 —— 脚本更快、更可靠，并且可以跨迭代重用。

2. **聚合到基准测试中** —— 运行技能创建器目录中的聚合脚本：
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <名称>
   ```
   这将生成 `benchmark.json` 和 `benchmark.md`，其中包含每个配置的通过率、时间和 Token，以及均值 ± 标准差和增量（delta）。如果手动生成 `benchmark.json`，请参见 `references/schemas.md` 以获取查看器预期的确切架构。
   将每个 `with_skill` 版本放在其对应的基准版本之前。

3. **进行分析员透视** —— 读取基准测试数据并发现聚合统计数据可能隐藏的模式。参见 `agents/analyzer.md`（“分析基准测试结果”部分）以了解要查找的内容 —— 例如无论是否有技能总是通过的断言（无辨别力）、高方差的评测（可能不稳定）以及时间/Token 权衡。

4. **启动查看器**，展示定性输出和定量数据：
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   对于迭代 2+，还要传递 `--previous-workspace <workspace>/iteration-<N-1>`。

   **协作/无头环境**：如果 `webbrowser.open()` 不可用或环境无显示，使用 `--static <输出路径>` 编写独立的 HTML 文件，而不是启动服务器。当用户点击“提交所有评审”时，反馈将作为 `feedback.json` 文件下载。下载后，将 `feedback.json` 复制到工作区目录中，以便下一次迭代提取。

   注意：请使用 `generate_review.py` 创建查看器；无需编写自定义 HTML。

5. **告诉用户** 类似这样的话：“我已经在你浏览器中打开了结果。有两个选项卡 —— ‘输出（Outputs）’ 让你点击查看每个测试用例并留下反馈，‘基准测试（Benchmark）’ 显示定量对比。完成后，请回到这里告知我。”

### 用户在查看器中看到的内容

“输出（Outputs）”选项卡一次显示一个测试用例：
- **Prompt**: 给出的任务
- **Output**: 技能产出的文件，尽可能内联渲染
- **Previous Output** (迭代 2+): 折叠部分，显示上一次迭代的输出
- **Formal Grades** (如果运行了评分): 折叠部分，显示断言通过/失败
- **Feedback**: 一个在输入时自动保存的文本框
- **Previous Feedback** (迭代 2+): 他们上次的评论，显示在文本框下方

“基准测试（Benchmark）”选项卡显示统计摘要：每个配置的通过率、计时和 Token 使用情况，以及每个评测的详细分解和分析员观察。

通过上一个/下一个按钮或方向键导航。完成后，他们点击“提交所有评审（Submit All Reviews）”，将所有反馈保存到 `feedback.json`。

### 第 5 步：读取反馈

当用户告诉你他们完成后，读取 `feedback.json`：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "图表缺少坐标轴标签", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完美，非常喜欢这个", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈意味着用户认为它没问题。将改进重点放在用户有具体抱怨的测试用例上。

完成后关闭查看器服务器：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进技能

这是循环的核心。你已经运行了测试用例，用户审查了结果，现在你需要根据他们的反馈改进技能。

### 如何思考改进

1. **从反馈中总结规律**。这里的大局是我们正在尝试创建可以被使用数百万次（甚至更多）的技能，适用于许多不同的提示词。在这里，你和用户只是在几个例子上反复迭代，因为它有助于加快进度。用户对这些例子了如指掌，评估新输出对他们来说很快。但如果你和用户共同开发的技能只适用于这些例子，那它就是无用的。与其进行繁琐的过度拟合修改，或者施加压迫性的约束性“必须（MUST）”，如果存在一些顽固问题，你可以尝试拓展思路，使用不同的比喻，或推荐不同的工作模式。尝试的成本相对较低，也许你会发现很棒的方法。

2. **保持提示词精简**。移除那些没有发挥作用的内容。确保阅读执行记录（transcripts），而不只是最终输出 —— 如果看起来技能让模型浪费了大量时间做一些无意义的事情，你可以尝试去掉技能中导致这些行为的部分，看看会发生什么。

3. **解释原因**。尽力解释你要求模型做的每一件事背后的**原因**。今天的 LLM 很“聪明”。它们有很好的心理理论，在获得良好的引导后，可以超越死板的指令，真正实现目标。即使用户的反馈很简短或令人沮丧，也要尝试真正理解任务、用户为什么写那些内容、他们到底写了什么，然后将这种理解转化为指令。如果你发现自己用大写字母写 ALWAYS 或 NEVER，或者使用超级僵硬的结构，那是一个警示信号 —— 如果可能，重新表述并解释原因，以便模型理解你所要求的事情为什么重要。这是一种更人性化、更强大且更有效的方法。

4. **寻找跨测试用例的重复工作**。阅读测试运行的执行记录，注意子 Agent 是否都独立编写了类似的辅助脚本或采取了相同的多步骤方法。如果所有 3 个测试用例都导致子 Agent 编写了 `create_docx.py` 或 `build_chart.py`，这是一个强烈的信号，表明技能应该捆绑该脚本。编写一次，放在 `scripts/` 中，并告诉技能去使用它。这可以节省未来每一次调用的重复劳动。

这项任务非常重要（我们正试图在这里创造巨大的经济价值！），你的思考时间不是瓶颈；花点时间好好思考。我建议先写一个修改草案，然后重新审视并进行改进。尽你最大的努力进入用户的头脑，理解他们的需求。

### 迭代循环

改进技能后：

1. 将改进应用到技能中。
2. 将所有测试用例重新运行到新的 `iteration-<N+1>/` 目录中，包括基准运行。如果你正在创建新技能，基准始终是 `without_skill`（无技能） —— 这在迭代过程中保持不变。如果你正在改进现有技能，根据你的判断选择合理的基准：用户最初带来的原始版本，或者是上一次迭代。
3. 启动查看器，`--previous-workspace` 指向上一次迭代。
4. 等待用户评审并告诉他们已完成。
5. 读取新反馈，再次改进，重复。

持续进行直到：
- 用户表示满意。
- 反馈全部为空（一切看起来都很好）。
- 你没有取得有意义的进展。

## 高级：盲测对比

对于你想要对技能的两个版本进行更严格对比的情况（例如，用户问“新版本真的更好吗？”），可以使用盲测对比系统。详情请阅读 `agents/comparator.md` 和 `agents/analyzer.md`。基本思路是：将两个输出交给一个独立的 Agent，不告诉它哪个是哪个，让它判断质量。然后分析获胜方获胜的原因。

这是可选的，需要子 Agent，大多数用户不需要它。人类评审循环通常就足够了。

## 描述优化

`SKILL.md` 前置元数据中的 `description` 字段是决定 Claude 是否调用技能的主要机制。在创建或改进技能后，提议优化描述以提高触发准确性。

### 第 1 步：生成触发评测查询

创建 20 个评测查询 —— 包含“应触发”和“不应触发”的混合。保存为 JSON：

```json
[
  {"query": "用户提示词", "should_trigger": true},
  {"query": "另一个提示词", "should_trigger": false}
]
```

查询必须是真实的，并且是 Claude Code 或 Claude.ai 用户实际会输入的内容。不是抽象的请求，而是具体、详细的请求。例如，文件路径、关于用户工作或情况的个人背景、列名和值、公司名称、URL。一些背景故事。有些可能是小写，或包含缩写、拼写错误或口语化表达。混合使用不同长度的查询，并专注于边界情况而非显而易见的情况（用户将有机会签字确认）。

差的例子：`"格式化此数据"`, `"从 PDF 提取文本"`, `"创建一个图表"`

好的例子：`"好的，我的老板刚才发给我这个 xlsx 文件（在我的下载里，叫类似 'Q4 sales final FINAL v2.xlsx' 之类的名字），她想让我增加一列显示利润率百分比。收入在 C 列，成本在 D 列，我想是这样的"`

对于**应触发**查询（8-10 个），考虑覆盖范围。你需要对同一意图使用不同的表述 —— 有些正式，有些随意。包含用户没有明确说出技能或文件类型但显然需要它的情况。加入一些不常见的用例，以及该技能与其他技能竞争但应胜出的情况。

对于**不应触发**查询（8-10 个），最有价值的是“近乎命中”的例子 —— 与技能共享关键字或概念但实际上需要其他东西的查询。考虑相邻领域、模糊表述（单纯的关键字匹配会触发但实际上不应该触发的情况），以及查询涉及技能所做的事情但在另一个工具更合适的情况。

关键要避免：不要让“不应触发”查询变得显而易见。对于 PDF 技能，“编写一个斐波那契函数”作为负面测试太容易了 —— 它没有测试任何东西。负面案例应该是真正棘手的。

### 第 2 步：与用户共同评审

使用 HTML 模板向用户展示评测集进行评审：

1. 从 `assets/eval_review.html` 读取模板。
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → 评测项的 JSON 数组（周围不加引号 —— 这是一个 JS 变量赋值）
   - `__SKILL_NAME_PLACEHOLDER__` → 技能名称
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 技能当前描述
3. 写入临时文件（例如 `/tmp/eval_review_<skill-name>.html`）并打开：`open /tmp/eval_review_<skill-name>.html`
4. 用户可以编辑查询、切换“应触发”状态、添加/删除条目，然后点击“导出评测集（Export Eval Set）”。
5. 文件下载到 `~/Downloads/eval_set.json` —— 检查下载文件夹中最近的版本，以防有多个版本（例如 `eval_set (1).json`）。

这一步很重要 —— 糟糕的评测查询会导致糟糕的描述。

### 第 3 步：运行优化循环

告诉用户：“这需要一些时间 —— 我将在后台运行优化循环并定期检查。”

将评测集保存到工作区，然后在后台运行：

```bash
python -m scripts.run_loop \
  --eval-set <触发评测 JSON 路径> \
  --skill-path <技能路径> \
  --model <支持本次会话的模型 ID> \
  --max-iterations 5 \
  --verbose
```

使用来自你系统提示词的模型 ID（支持当前会话的那个），以便触发测试与用户的实际体验相匹配。

在运行期间，定期查看输出以向用户更新当前处于第几次迭代以及得分情况。

这会自动处理完整的优化循环。它将评测集分为 60% 训练集和 40% 留出测试集，评估当前描述（每个查询运行 3 次以获得可靠的触发率），然后调用 Claude 根据失败情况提出改进建议。它在训练集和测试集上重新评估每个新描述，最多迭代 5 次。完成后，它在浏览器中打开一个 HTML 报告，显示每次迭代的结果，并返回带有 `best_description` 的 JSON —— 根据测试得分而非训练得分选择，以避免过度拟合。

### 技能触发如何工作

理解触发机制有助于设计更好的评测查询。技能以名称 + 描述的形式出现在 Claude 的 `available_skills` 列表中，Claude 根据该描述决定是否查阅技能。重要的是要知道，Claude 只对它无法轻易独立处理的任务查阅技能 —— 简单的单步查询（如“读取此 PDF”）即使描述完全匹配也可能不会触发技能，因为 Claude 可以使用基础工具直接处理。复杂、多步或专门的查询在描述匹配时会可靠地触发技能。

这意味着你的评测查询应该具有足够的实质内容，使得 Claude 确实能从查阅技能中受益。像“读取文件 X”这样的简单查询是糟糕的测试用例 —— 无论描述质量如何，它们都不会触发技能。

### 第 4 步：应用结果

从 JSON 输出中获取 `best_description` 并更新技能的 `SKILL.md` 前置元数据。向用户展示修改前后的对比并报告得分。

### 打包并展示（仅当 `present_files` 工具可用时）

检查你是否可以使用 `present_files` 工具。如果没有，跳过此步。如果有，打包技能并将 `.skill` 文件展示给用户：

```bash
python -m scripts.package_skill <技能文件夹路径>
```

打包后，引导用户到生成的 `.skill` 文件路径，以便他们安装。

## Claude.ai 特定说明

在 Claude.ai 中，核心工作流是相同的（起草 → 测试 → 评审 → 改进 → 重复），但由于 Claude.ai 没有子 Agent，一些机制会有所改变。以下是需要调整的地方：

**运行测试用例**：没有子 Agent 意味着无法并行执行。对于每个测试用例，阅读技能的 `SKILL.md`，然后按照其指令亲自完成测试提示词。一次做一个。这比独立的子 Agent 严谨性稍差（你是技能编写者，同时也在运行它，所以你拥有完整的上下文），但它是一个有用的完整性检查 —— 人类评审步骤可以弥补这一点。跳过基准运行 —— 只需按要求使用技能完成任务。

**评审结果**：如果你无法打开浏览器（例如 Claude.ai 的虚拟机没有显示器，或者你在远程服务器上），请完全跳过浏览器查看器。相反，直接在对话中展示结果。对于每个测试用例，展示提示词和输出。如果输出是用户需要查看的文件（如 .docx 或 .xlsx），将其保存到文件系统并告诉他们位置，以便他们下载和检查。内联询问反馈：“这个看起来怎么样？有什么需要修改的吗？”

**基准测试**：跳过定量基准测试 —— 它依赖于在没有子 Agent 的情况下没有意义的基准对比。专注于用户的定性反馈。

**迭代循环**：与之前相同 —— 改进技能，重新运行测试用例，请求反馈 —— 只是中间没有浏览器查看器。你仍然可以在文件系统上按迭代目录组织结果。

**描述优化**：此部分需要 `claude` CLI 工具（特别是 `claude -p`），该工具仅在 Claude Code 中可用。如果你在 Claude.ai 上，请跳过此步。

**盲测对比**：需要子 Agent。跳过。

**打包**：`package_skill.py` 脚本在任何有 Python 和文件系统的地方都可以运行。在 Claude.ai 上，你可以运行它，用户可以下载生成的 `.skill` 文件。

**更新现有技能**：用户可能会要求你更新现有技能，而不是创建新技能。在这种情况下：
- **保留原始名称**。记录技能的目录名称和 `name` 前置元数据字段 —— 保持原样使用。例如，如果安装的技能是 `research-helper`，输出 `research-helper.skill`（而不是 `research-helper-v2`）。
- **在编辑前复制到可写位置**。安装的技能路径可能是只读的。复制到 `/tmp/skill-name/`，在那里编辑，然后从副本进行打包。
- **如果手动打包，先在 `/tmp/` 中暂存**，然后复制到输出目录 —— 由于权限问题，直接写入可能会失败。

## 协作 (Cowork) 特定说明

如果你在协作（Cowork）环境中，主要需要了解以下几点：

- 你有子 Agent，因此主工作流（并行启动测试用例、运行基准测试、评分等）都可以运行。（但是，如果你遇到严重的超时问题，可以串行而不是并行运行测试提示词。）
- 你没有浏览器或显示器，因此在生成评测查看器时，使用 `--static <输出路径>` 编写独立的 HTML 文件，而不是启动服务器。然后提供一个链接，用户可以点击该链接在他们的浏览器中打开 HTML。
- 无论出于何种原因，协作环境似乎不太倾向于在运行测试后生成评测查看器，因此再次重申：无论你是在协作环境还是在 Claude Code 中，在运行测试后，你应该始终生成评测查看器，以便人类在你自己修订技能并尝试更正之前查看示例，使用 `generate_review.py`（而不是编写你自己的精品 HTML 代码）。提前道歉，但我要在这里用大写字母：在你自己评估输入 *之前* 生成评测查看器。你想尽快把它们展示在人类面前！
- 反馈的工作方式不同：由于没有运行中的服务器，查看器的“提交所有评审”按钮将反馈下载为 `feedback.json` 文件。然后你可以从那里读取它（你可能需要先请求访问权限）。
- 打包可以运行 —— `package_skill.py` 只需要 Python 和文件系统。
- 描述优化（`run_loop.py` / `run_eval.py`）在协作环境中应该可以正常运行，因为它通过子进程使用 `claude -p` 而非浏览器，但请等到你完全完成技能制作且用户同意其处于良好状态后再进行。
- **更新现有技能**：用户可能会要求你更新现有技能，而不是创建新技能。遵循上述 Claude.ai 部分中的更新指南。
