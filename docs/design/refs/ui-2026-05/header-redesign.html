方案 A 的设计决策

  • 面包屑只占 36px，所有导航上下文压缩到一行；linux-disk-cleanup v0 放在面包屑末端作为"当前 skill"指示，旁边的 ⇅ 图标提供切换入口。
  • #12 改进 prompt v2 后回归 作为 H1（18px / 600），占据主视线焦点；#12 用等宽字体 + 灰底胶囊，沿用 GitHub Issue 的语义。
  • "历史 / 新建任务 / 保存配置 / 更多"全部降级为 30×30 的 icon button，hover 显示 tooltip。历史按钮右上角带 8 的数字气泡，保留信息量。
  • 只有「开始评测」是填色主按钮，左侧用 0.5px 分隔线和 icon 组隔开，避免误触。
  • 元数据（时间 / 用例数 / 版本）独立成一行小字，12px / text-tertiary，是参考信息而非决策信息。
  • LLM Judge 标签从原本的「用例分析」标题旁移到 header 元数据右端，作为任务属性而非页面属性更合理。

方案 B 的设计决策

  • 第一层是 40px 的灰底 context strip：左边是简化的面包屑（不含 skill），右边是常驻的 skill 切换器 chip。skill 切换变成了一个明确的、视觉上独立的入口。
  • 第二层是任务主体区，H1 标题 + 状态徽章 + LLM Judge 标签作为身份信息；下方一行左边是元数据，右边是动作组。"历史"作为可点击的元数据展示8 个版本，比单独按钮更有信息量。
  • 动作分两级：「新建任务」「保存配置」是线框次按钮，「开始评测」是填色主按钮，视觉权重一眼分明。
  • 适合场景：用户需要频繁在不同 skill 间对比评测任务（"linux-disk-cleanup 和 docker-cleanup 的 baseline 哪个好"）。
  • 代价：比方案 A 高 ~12px，但换来 skill 切换的常驻可见性。

当前实现的问题

  • 信息层级混乱：第二行把"评测任务"label、任务 ID、标题、状态、元数据、4 个动作按钮全塞在一行。
  • 主体被次要化：linux-disk-cleanup（skill）放在第一行末端，而真正的页面主体「#12 改进 prompt v2 后回归」被挤在第二行更小的位置。
  • 动作缺乏分组：「历史」（查看）、「新建任务」（创建）、「保存配置」（写当前态）、「开始评测」（触发流程）四个语义完全不同的按钮视觉权重几乎一致，"保存配置"和"开始评测"挨着尤其容易误触。
  • "评测任务"前缀冗余：#12 这种 ID 风格本身就足够表明这是一个任务，类似 GitHub Issue / Linear 的惯例。
